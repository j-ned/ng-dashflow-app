### Vue d'ensemble

`dash-flow` est une **PWA Angular 22 zoneless, mobile-first**, application familiale à **deux domaines vivants** : budget (enveloppes, prêts, comptes, transactions, récurrences, archives de salaire, analytics) et médical (patients, praticiens, rendez-vous, ordonnances, médicaments, documents, rappels), plus `auth`, `settings`, `admin` et une `landing` publique. Elle est organisée en **Clean Architecture EAK 3 couches par feature** (`domain/` TS pur → `infra/` HTTP+adapters → pages/composants smart/dumb), avec transverses `core/` (crypto E2EE, api, csrf, i18n, interceptors, guards), `layout/` (shells), `shared/` (primitives UI, validators). L'état partagé passe par des **stores signals maison** (pas de NgRx) ; les données sensibles sont **chiffrées end-to-end** (WebCrypto, `crypto.store`) et validées à la lecture via `zod/mini` avec garde-fou anti-dérive schéma↔type. Le backend REST NestJS vit dans un **repo séparé** (`nest-dashflow-app`) ; le dev pointe `apiUrl` en absolu sur `:3001`. **Mode de rendu : SPA pure** (aucune config SSR/SSG/prerender dans `angular.json`, `@angular/build:application` sans `server`/`prerender`, `provideClientHydration` absent) — aucun constat SEO/`<head>`/JSON-LD n'est donc soumis à un contrôle prerender ; la landing publique s'affiche en client-only.

La codebase est **exceptionnellement disciplinée** : **0 `any`** (specs incluses), **0 pattern legacy** (`*ngIf`/`NgModule`/`@ViewChild`/`*.component.ts` = zéro), **OnPush explicite sur 104/104 composants**, **0 dépendance cyclique** (madge), tous les `subscribe()` en `takeUntilDestroyed`, tous les listeners/timers nettoyés via `DestroyRef`. Les entry points sont `main.ts` → `app.config.ts` (providers, Sentry conditionnel) → routing lazy par feature. Les **hotspots** (volume × churn) sont concentrés dans les pages budget, dominés par `bank-account.ts` (977 LOC, 47 commits/6 mois — 2× le second). La dette n'est pas dans la qualité micro (irréprochable) mais dans **l'altitude des pages** (cohorte de god files), **l'absence de CI exécutée**, et une **dérive documentaire** (profil + README).

### Bonnes pratiques notables (optionnel, ≤ 5 entrées — informatif)

- **Hygiène ressources exemplaire** — Tous les `subscribe()` (bank-account, transactions, admin-page, csv-import-wizard) passent `takeUntilDestroyed` ; `theme.store`/`locale.store` retirent leurs listeners via `DestroyRef.onDestroy` ; le rollover timer de `medical-dashboard` est `clearTimeout` à la destruction. Aucune fuite détectée. À reproduire.
- **Garde-fou schéma↔type anti-dérive** — Chaque schéma `zod/mini` porte `_Check = Expect<MutualAssign<z.infer<Schema>, Model>>` : le build casse si le schéma diverge du `type` domaine. Excellent contrat compilé.
- **Coercition frontière isolée en adapter pur** — Le footgun `numeric` Postgres (string en clair vs number chiffré) est traité dans des fonctions pures `*.adapter.ts` avec WHY documenté, sans polluer le schéma (`z.number()` conservé). Pattern propre et testable.
- **`eslint-disable` toujours justifiés** — Chaque désactivation (native `<dialog>`, images `data:`/distantes sans `IMAGE_LOADER`) porte un WHY précis en une ligne. Discipline anti-suppression-aveugle.
- **Typage strict tenu** — 0 `any` sur 366 fichiers specs incluses, OnPush explicite 104/104 (correct sur projet migré v22), 0 cycle de dépendances.
